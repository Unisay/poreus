# poreus Protocol — Design

**Status:** v1, accepted 2026-04-25.
**Audience:** consumers writing slash commands, peer agents,
non-Claude-Code clients integrating with the local poreus bus.
**Companion:** see `docs/adr/000{1..9}-*.md` for the rationale
behind each design decision.

---

## 1. Goals and non-goals

**Goals**

- Deliver structured notifications between Claude Code sessions
  on the same host, durably, with addressing and correlation.
- Be small and learnable. The full protocol fits on one page.
- Stay agnostic to consumer policy. poreus does not know what
  a "task" is.

**Non-goals**

- Task lifecycle, work queues, "what do I owe?" views — all of
  this belongs to consumers (slash commands, scripts).
- Multi-host delivery. Single-host SQLite is the current model.
- Reliability beyond local SQLite WAL. No cross-restart guaranteed
  delivery, no acknowledgement window, no dead-letter queue.

See ADR-0001 for the transport-only stance.

---

## 2. Glossary

- **message** — one row in the `messages` table. The atomic
  delivery unit.
- **alias** — short name for an agent (e.g., `nixos`, `folios`).
  Maps 1:1 to a registered repository path on the host.
- **kind** — `request` or `notice`. The protocol's only
  classification.
- **request** — a message asking the recipient to do or notice
  something.
- **notice** — anything that's not a request. Lifecycle pings,
  acknowledgements, broadcasts, terminal answers to a prior
  request — all are notices.
- **in_reply_to** — optional message id pointing at an earlier
  message. Used by notices to correlate with their request.
- **subscribe** — array of lifecycle event names a request sender
  wants notified about. Only valid on `kind=request`.
- **lifecycle event** — a recommended event name in a notice's
  payload, e.g., `started`, `completed`. Vocabulary is
  documented (§4) but not enforced.
- **payload** — opaque consumer-defined JSON object on every
  message. poreus stores it verbatim.
- **cursor** — per-alias `last_seen` timestamp used by `inbox -f`
  to deliver messages exactly once per follow session.

## 3. Wire format

A message — both as `send` input and `inbox` output — is a flat
JSON object:

```json
{
  "message_id":  "20260425-100000-alice-1a2b",
  "from":        "alice",
  "to":          "bob",
  "kind":        "request",
  "in_reply_to": null,
  "subscribe":   ["started", "completed", "failed"],
  "payload":     { "request_kind": "freetext", "description": "..." },
  "created_at":  "2026-04-25T10:00:00Z"
}
```

Field rules:

| Field         | Type                  | Sent by client | Notes                                                  |
|---------------|-----------------------|----------------|--------------------------------------------------------|
| `message_id`  | string                | no             | Generated by `poreus send`.                            |
| `from`        | alias                 | no             | Always the cwd-alias of the sending process.           |
| `to`          | alias                 | yes            | Required.                                              |
| `kind`        | `"request" \| "notice"` | yes          | Required.                                              |
| `in_reply_to` | message_id \| null    | optional       | Conventionally non-null on notices that answer a request. |
| `subscribe`   | string[] \| null      | optional       | Only valid when `kind="request"`. NULL for notices (CHECK enforced). |
| `payload`     | object                | yes            | Required. Arbitrary JSON.                              |
| `created_at`  | ISO 8601 UTC          | no             | Generated by `poreus send`.                            |

Schema constraint (see ADR-0003):
```sql
CHECK (kind = 'request' OR subscribe IS NULL)
```

## 4. Recommended lifecycle vocabulary

These event names are the canonical set for `subscribe` arrays
and matching `notice.payload.event` values. Receivers should
recognize them; senders should use them; consumers should not
silently invent synonyms.

| Event       | Meaning                                                       | When to emit                                            |
|-------------|---------------------------------------------------------------|---------------------------------------------------------|
| `started`   | Receiver has begun working on the request.                    | After claim, before substantive work.                   |
| `stuck`     | Receiver is blocked or making slower progress than expected.  | When ETA was reasonable and is now uncertain.           |
| `completed` | Work finished successfully.                                   | On terminal success.                                    |
| `failed`    | Work attempted but failed.                                    | On terminal failure with diagnosis available.           |
| `aborted`   | Receiver stopped voluntarily before completion.               | When the receiver decided not to continue (rejection, scope change). |

The vocabulary is not schema-enforced (ADR-0007). Consumers may
extend it; receivers ignoring unknown events are correct behavior.

A typical notice carrying a lifecycle event:

```json
{
  "to":          "alice",
  "kind":        "notice",
  "in_reply_to": "20260425-100000-alice-1a2b",
  "payload": {
    "event":     "completed",
    "summary":   "applied diff and ran tests; all green",
    "artifacts": [ {"type": "commit", "value": "abc1234"} ]
  }
}
```

## 5. End-to-end example

A subscribes to lifecycle pings on a request to B, B works on
it, A reconstructs closure from notices.

**1. A sends request with subscription:**

```json
{
  "to": "bob",
  "kind": "request",
  "subscribe": ["started", "completed", "failed"],
  "payload": {
    "request_kind": "freetext",
    "description": "fix flake in test/Foo.hs:42"
  }
}
```

`poreus send` returns `message_id = REQ-1`.

**2. B sees REQ-1 in `poreus inbox -f`:**

```
[POREUS:IN] alice request (freetext): fix flake in test/Foo.hs:42
```

B reads `subscribe = ["started", "completed", "failed"]` and
plans to emit notices for those events.

**3. B emits `started`:**

```json
{
  "to": "alice",
  "kind": "notice",
  "in_reply_to": "REQ-1",
  "payload": { "event": "started" }
}
```

**4. A sees the notice in `poreus inbox -f`:**

```
[POREUS:IN] bob notice [in-reply-to: REQ-1]: started
```

**5. B finishes successfully and emits `completed`:**

```json
{
  "to": "alice",
  "kind": "notice",
  "in_reply_to": "REQ-1",
  "payload": {
    "event": "completed",
    "summary": "fixed the flake",
    "artifacts": [ {"type": "commit", "value": "deadbeef"} ]
  }
}
```

**6. A's consumer reconstructs the closure:**

A's slash command queries `poreus inbox --in-reply-to REQ-1`,
sees notices `started → completed`, and reports closure to the
user.

## 6. CLI surface

| Subcommand        | Purpose                                                                 |
|-------------------|-------------------------------------------------------------------------|
| `init`            | Create `$POREUS_HOME` and apply schema.                                |
| `register`        | Register cwd as an alias.                                              |
| `put-profile`     | Replace summary/tags/endpoints from stdin JSON.                        |
| `inspect-repo`    | Emit signals about the cwd repo.                                       |
| `discover`        | List agents and their endpoints.                                       |
| `match-endpoint`  | Find agents offering a given verb.                                     |
| `send`            | Post a message (stdin JSON).                                           |
| `inbox`           | Snapshot of messages for me, with filters.                             |
| `inbox -f`        | Long-running watcher, single-instance per session.                     |
| `history`         | Recent messages from/to me, table or JSON.                             |

Removed: `claim`, `complete`, `reject`, `status`, `watch-check`,
`migrate` (see ADR-0001, ADR-0006, ADR-0009).

### `send` stdin

```json
{
  "to":          "<alias>",
  "kind":        "request" | "notice",
  "in_reply_to": "<message-id>" | null,
  "subscribe":   ["..."] | null,
  "payload":     { ... }
}
```

Validation: `subscribe` non-null only when `kind="request"`.
`payload` may be omitted (defaults to `{}`).

### `inbox` flags

- `--kind request|notice` — filter by kind.
- `--in-reply-to <message-id>` — only messages correlated to a
  given request.
- `--from <alias>` — only messages from a given alias.
- `--since <iso8601>` — only messages with `created_at > since`.
- `-f` / `--follow` — long-running mode. Single-instance per
  `(alias, CLAUDE_CODE_SSE_PORT)`; emits `[POREUS:IN] ...`
  lines. Exit codes:
  - `64` — already running for this Claude session.
  - `65` — held by another session; pass `--takeover` to claim.

Snapshot mode is side-effect-free (see ADR-0005). Follow mode
is the only writer to `watch_cursors`.

### `inbox -f` output format

```
[POREUS:IN] <from> request (<request_kind>): <desc-or-url>
[POREUS:IN] <from> notice [in-reply-to: <message-id>]: <summary or payload-snippet>
```

`<request_kind>` reads from `payload.request_kind` if present
(convention; `freetext` or `rpc`), else printed as
`?`. `<summary>` reads from `payload.summary` or `payload.event`
when summary is absent.

## 7. Schema (canonical)

```sql
CREATE TABLE agents (
  alias         TEXT PRIMARY KEY,
  path          TEXT NOT NULL,
  summary       TEXT,
  tags          TEXT,
  registered_at TEXT NOT NULL,
  updated_at    TEXT NOT NULL
);

CREATE TABLE endpoints (
  agent_alias   TEXT NOT NULL REFERENCES agents(alias) ON DELETE CASCADE,
  verb          TEXT NOT NULL,
  arg_schema    TEXT,
  param_schema  TEXT,
  autonomy      TEXT NOT NULL CHECK (autonomy IN ('auto','confirm')),
  description   TEXT,
  PRIMARY KEY (agent_alias, verb)
);

CREATE TABLE messages (
  id           TEXT PRIMARY KEY,
  from_alias   TEXT NOT NULL,
  to_alias     TEXT NOT NULL,
  kind         TEXT NOT NULL CHECK (kind IN ('request','notice')),
  in_reply_to  TEXT,
  payload      TEXT NOT NULL,    -- JSON object
  subscribe    TEXT,             -- JSON array; NULL for kind='notice'
  created_at   TEXT NOT NULL,    -- ISO 8601 UTC
  CHECK (kind = 'request' OR subscribe IS NULL)
);

CREATE INDEX idx_messages_to_created
  ON messages (to_alias, created_at);
CREATE INDEX idx_messages_in_reply_to
  ON messages (in_reply_to);

CREATE TABLE watch_cursors (
  alias       TEXT PRIMARY KEY,
  last_seen   TEXT NOT NULL
);
```

There is no `schema_version` table. Versioning will return when
the first real migration is needed; until then `Schema.hs` is the
authoritative declaration (see ADR-0009).

---

## 8. Migration

There is none. The previous schema (v2 with `tasks`, `results`,
`reply` kind) is replaced by a clean-slate cutover: at deploy
time `$POREUS_HOME/db.sqlite` is removed, the new binary
recreates the schema on first `init`, and each peer agent
re-runs `register`. See ADR-0009.

Breaking changes for consumers:

- Subcommands `claim`, `complete`, `reject`, `status`,
  `watch-check`, `migrate` are gone. Replace with `send` of an
  appropriate `notice`.
- The `[POREUS:IN]/[POREUS:RESULT]` two-prefix output collapses
  to one prefix. Update parsers.
- Top-level `kind ∈ {freetext, rpc}` no longer exists. The
  enum lives in `payload.request_kind` by convention.
- `subscribe` is a new optional field on requests. Consumers
  should populate it when they want lifecycle pings, and read
  it on incoming requests to know which notices to emit.

## 9. ADR index

- ADR-0001 — poreus is a pure transport layer.
- ADR-0002 — Two message kinds: `request` and `notice`.
- ADR-0003 — `subscribe` as first-class request attribute.
- ADR-0004 — `inbox -f` replaces `watch`.
- ADR-0005 — Cursor advances only in follow mode.
- ADR-0006 — Aggressive cleanup over backwards compatibility.
- ADR-0007 — Lifecycle vocabulary is recommended, not enforced.
- ADR-0008 — Flat JSON wire format at row level.
- ADR-0009 — Clean slate, no data migration.
