# ADR-0009: Clean slate, no data migration

## Status

Accepted — 2026-04-25.

## Context

The redesign changes the schema substantially:

- Drop `tasks`, `results`, `schema_version`.
- Update `messages.kind` CHECK constraint.
- Add `messages.subscribe` column with cross-column CHECK.
- Reset table set to `agents`, `endpoints`, `messages`,
  `watch_cursors`.

Schema migration in SQLite for CHECK changes requires the
`CREATE-COPY-DROP-RENAME` dance. It is ~30 lines of SQL plus
test fixtures, with edge cases for partial-state databases.

The user's existing database holds:
- Live messages (pending requests, in-flight notices).
- Agent registrations and endpoint declarations.
- Watch cursors.

All of these can be regenerated by re-running `poreus init`,
`poreus register`, and `put-profile` from each agent. The cost
is ~30 seconds of human time per peer.

## Decision

Skip migration entirely. At cutover:

1. The new binary deploys with no awareness of v2 data.
2. The cutover script `rm -f $POREUS_HOME/db.sqlite`.
3. The next `poreus init` creates a fresh schema from the
   declarative `Schema.hs`.
4. Each peer re-registers its profile.

`Schema.hs` becomes a flat set of `CREATE TABLE IF NOT EXISTS`
statements with no version tracking. Versioning returns when
the first real migration is needed (probably never, given the
single-host scope).

## Consequences

**Positive**

- ~30 lines of SQL and a class of test fixtures disappear from
  the codebase.
- Schema documentation becomes literal — read `Schema.hs` and
  you have the truth.
- No risk of mid-migration corruption or partial-state DBs.
- Subcommand `migrate` (legacy a2a-importer) goes too — already
  obsolete since 2026-Q1.

**Negative**

- All in-flight tasks at cutover time are lost (already
  accepted under ADR-0006).
- Peers must manually `poreus init && /poreus:register` after
  cutover. Documented in the cutover runbook.
- Future redesigns of similar magnitude won't have a migration
  precedent to copy from.

## Alternatives Considered

- **Full migration via CREATE-COPY-DROP-RENAME.** Standard,
  works, well-understood. Rejected because the data being
  preserved (in-flight tasks at the moment of deploy) has
  near-zero half-life — peers would have to retry anyway.
- **Partial migration: keep `agents`/`endpoints`, drop
  `tasks`/`results`, transform `messages`.** Saves the
  re-registration step. Saving 30 seconds of human time isn't
  worth the migration code; rejected.
